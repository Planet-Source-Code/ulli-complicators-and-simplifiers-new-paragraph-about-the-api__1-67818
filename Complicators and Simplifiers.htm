Option Explicit

 

Use it, use it, use it! There is an option under Extras|Options in the Editor tab – it’s called Variable Declaration Required and if you checkmark that VB will automatically insert that line for you. If this option is on VB will force you to declare all variables by name (unfortunately not by scope and type also)  and will check that the name is declared when you use it in your code. It will even correct the case (upper or lower) for you. Without his option variables are declared when you first use them and very probably not by the type that is most appropriate; and misspelling a variable name goes by unnoticed by VB and will cause you a lot of trouble locating where VB used the misspelled variable instead of the one you meant.

 

Variables, Constants

 

One per line, always with type (not type suffix) and scope, constants too have a type and a scope. And Integers are out since we live in a 32-bit world and memories are huge; the unit of transfer between memory and CPU is 32 bits, shorter values will have to be masked and perhaps shifted into place, making them slower to handle. This is particularly true for the  As Byte type.

 

Local variables (Dim) are declared at the top of each module; these are not procedural statements and putting them into any kind of condition does not keep them from being allocated on the stack.  What “the Stack” is will be explained further down.

 

If Name = “Ulli” then

    Dim MiddleName As String

    MiddleName = “Theophrastos”

End If

 

MiddleName is declared and allocated  anyway, not just for people called Ulli; the reader of your code however may not notice that a local variable of that name exists in the module….

 

   Dim i As Long

  

   For i = 1 To 100

        Dim j As Long

        j = j + 10

   Next i

 

…nor will it allocate a table or array, or reset the value to it’s default during each iteration. 

 

If you squeeze more than one variable declaration into one line then make sure that each and everyone is declared  As [Type] because VB does not (repeat NOT) behave like Pascal or C where one type declaration is valid for all preceding names; in VB all but the last would in fact be defaulted to  Variants.

 

Do not use Variants unless you have a very good reason to use them. Un-typed variables (and Functions, see below) default to Variants (see above) and these will be coerced (forcibly converted) into the proper type every time you use them, making your code unnecessarily slow.

Subs, Functions, Properties, Enums, Types and Event Declarations

 

They also have a scope, and Functions and Property Get’s have a type. Do not use a  Function unless it returns a value. Inside the function it’s name can be used just like a variable, you don’t have to assign the function value to a variable first and the assign that to the  Function.

 

Given:

 

Private Type tPoint

    X           As Long

    Y           As Long

End Type

 

 

Private Type tRect

    TopLeft     As tPoint

    BottomRight As tPoint

End Type

 

Private MyRect  As tRect

 

Private Function MakePoint(newX As Long, newY As Long) As tPoint

 

    With MakePoint

        .x = newX

        .y = newY

    End With

 

End Function

 

 

Private Function MakeRect (l As Long, t As Long, r As Long, b As Long) As tRect

 

    With MakeRect

        .TopLeft = MakePoint(l, t)

        .BottomRight = MakePoint(r, b) 

    End With

 

End Functiom

 

then

 

MyRect = MakeRect(17, 44, 64, 88)

 

will create a rectangle with the four corners at coordinates  (17, 44),  (17, 88),  (64, 88),  (64, 44). Take your time now to work out why and how this is so and understand the principle behind it.

 

 

Numeric API Constants

 

It’s a good idea to put them all in an Enumeration

 

Private Enum ApiConst

    WM_KEYDOWN = &H100

    WM_KEYUP = &H101

    WM_CHAR = &H102

    WM_HSCROLL = &H114

    WM_VSCROLL = &H115

    WM_LBUTTONUP = &H202

    WM_MBUTTONDOWN = &H207

    WM_MOUSEWHEEL = &H20A

    WM_MDIACTIVATE = &H222

End Enum

 

That makes them  Long  (the type the API is expecting in most cases) because all variables inside an  Enum are  Long by default. Enums have the nasty effect however not to be case-persistent; when you type them differently (different upper/lower case) then that will change the declaration. A “bizarre hack”, as Vlad called it, will overcome that VB-bug though; repeat the names inside a false conditional compilation bracket:

 

#If False Then ‘this will not be compiled

Private WM_KEYDOWN, WM_KEYUP, WM_CHAR, WM_HSCROLL, WM_VSCROLL, WM_LBUTTONUP, WM_MBUTTONDOWN, WM_MOUSEWHEEL, WM_MDIACTIVATE

#End If

 

Ulli’s Code Formatter or  Roger’s Code Fixer will do that for you.

 

What “the API” is will be explained in later tutorial.

 

Conditions

 

VB has an expression evaluator (a piece of coding inside VB to work out what the result of an expression is) and every expression is presented to it. It evaluates the expression and returns an intermediate result of a type deemed best according to the type of variables present in the expression. This result is then presented to the receiver; this may be a variable, a property, an object or it may be the If function itself. Some receivers accept the result as it is, with others there is a type coercion necessary. In case of If it is coerced to a Boolean type unless it is already a  Boolean. Variables which can be compared to True or False are already Booleans,  comparisons however return Booleans; so there is no need to compare them in the first place.

 

If Control.Enabled = True Then is a pleonasm  and it is sufficient enough to write If Control.Enabled Then

 

As a matter of fact every numeric variable is also coerced to the receiver type if necessary and in the case of If a numeric will be coerced to a Boolean. Remember that zero is False and everything else is True. So any numeric variable can also be presented to an If and is coerced accordingly.

The opposite is also true, any Boolean (where True is represented by  –1 and False by 0) can be coerced to a numeric type, even a Boolean returned from the expression evaluator, so

 

i = i – (i < 3)

 

will subtract  ( –1)  from i  (that is: add 1)  if i is less than 3  and do nothing if not.

 

The reason for these truth values by the way is that the  Not operator switches all bits to the opposite state and therefore 0 becomes  –1 and vice versa, making  Not True = False and  Not False = True.

 

Here is another example:

 

Sub Something

 

    If a <> 3 Then Exit Sub

    Code

    More Code

 

End Sub

 

 If you reverse the condition the code will look much nicer:

 

Sub Something

 

    If a = 3 Then

        Code

        More Code

    End If

 

End Sub

 

Always use the multiline version of  If – Then - Else – End If

 

If SomeCondition Then

    TruePart   

  Else

    FalsePart

End If

 

If a conditional statement becomes too complicated remember there is the Select – Case – End Select  structure. And that’s much more flexible than If, because the Case allows single values, multiple values, value ranges, equalities and inequalities and a mixture of these. Each  comma represents an implicit  Or

 

  • Case 3

 

  • Case 4, 5, 7

 

  • Case 9 To 11

 

  • Case 9 To 11, 24 To 27

 

  • Case 22, 24, 33 To 44

 

  • Case Not 24 To 27

 

  • Case Not 63, 64, 68 To 73

 

  • Case Is < 55, 58 To 63

 

  • Case Is >= 88

 

  • Case Is <> 90

 

  • Case Else

 

are all legal syntax, and code like

 

If SomeVariable = 1 Or _

   SomeVariable = 3 Or _

   SomeVariable = 5 Or _

   SomeVariable > 6 And SomeVariable < 11 Then

    do something

  Else

    do some other thing

End If

 

can  be replaced by

 

Select Case SomeVariable
  Case 1, 3, 5, 7 To 10

    do something

  Case Else

    do some other thing

End Select

 

 

Conditional expressions, like any other expression, have an operator precedence and you should use brackets only to alter the normal precedence ; And  and  Or are operators and  And  precedes  Or. If – in rare cases - you think that dispensable brackets could clarify the meaning of a conditional expression then use them for the sake of readability.

 

There is no short-circuiting in VB, meaning that the whole Boolean expression is always evaluated, even if it is clear before the end of the expression what the final result of the evaluation will be:

 

Given  Var1 = 3 and  Var2 = 5

 

If Var1 = 4 And Var2 = 18 then

 

After evaluating Var1 it is clear that the final result will be  False, Var2 however will nevertheless also be evaluated. If you want to omit unnecessary evaluations change the code to:

 

If Var1 = 4 Then

    If Var2 = 18 Then

 

 

Return Values of API Calls and Intrinsic Functions

 

Almost all API calls return a value but most of the time you don’t need that. Even if the prototype is defined as a function you would use it like you would use a Sub.

 

Public Declare Function SendMessage Lib "user32" Alias "SendMessageA" (ByVal hWnd As Long, ByVal wMsg As Long, ByVal wParam As Long, lParam As Any) As Long

 

SendMessage hWnd, nMsg, wParam, ByVal lParam

 

is legal in spite of the fact that SendMessage is declared as function.

 

If you have to use the return value of a function in another function then chain them:

 

Private Declare Function LoadCursor Lib "user32" Alias "LoadCursorA" (ByVal hInstance As Long, ByVal lpCursorName As Long) As Long ‘returns hCursor

 

Private Declare Function SetCursor Lib "user32" (ByVal hCursor As Long) As Long ’needs hCursor

 

and

 

hCursor = LoadCursor(0, CsrArrow)

SetCursur hCursor

 

can be abbreviated to

 

SetCursor LoadCursor(0, CsrArrow)

 

 

Still  another example where the return value of the  MsgBox-function is chained to a conditional expression whose result is presented to the If

 

RetVal = MsgBox(……..)

If RetVal = something Then…

 

could be shortened to

 

If MsgBox(…..) = Something Then…

 

and it has the advantage that VB will suggest names for the possible values.

As with all  Functions the  MsgBox Function has a  Type (same as with variables or constants) and in this case the type is  VbMsgBoxResult (which is derived from an intrinsic enumeration).

 

 

String-returning Intrinsic Functions

 

come in two flavors, one without he $-suffix and one with the $-Suffix. Use the latter ones, always, as the former ones return  Variants  which will most certainly have to be coerced.

 

 

Loops

 

For… Next

 

Repeat the For-variable with the Next statement.

 

Nested  For loops have an abbreviated  Next statement if the both end at the same place

 

For i = 1 To 10

    For j = 1 to 10

        [loop body]

Next j, i

 

If you modify the For-variable within the loop body be sure what you are doing. At the end of a  For loop the For-variable is one step higher (or lower in the case of a negative Step) than the To-value.

 

You can break a  For-loop by the  Exit For statement in which case the For-variable is left at it’s current value. So at the end of a loop you can test how the loop was terminated by comparing the For-variable to the To-value;  If For-variable   <=  To-value then the loop was broken, otherwise it was terminated normally.

 

Do… Loop

 

comes in a variety of flavors:

 

Unconditional

 

You have to provide the Exit condition somewhere within the loop yourself

 

Conditional Entry

 

Do While SomeCondition or Do Until SomeConditon

 

    [loop body]

 

Loop

 

The loop is only  entered if  the condition is True (While) or False (Until) and is then repeated until the condition is False (While) or True (Until)

 

Conditional Exit

 

Do

 

    [loop body]

 

Loop While SomeCondition or Loop Until SomeCondition

 

The loop is entered and repeated until the condition is False (While) or True (Until)

 

Combinations of the above

 

Do While SomeCondition or Do Until SomeConditon

 

    [loop body]

 

    If SomeBreakCondition Then

        Exit Do

    End If

 

Loop

 

 

Do not use While..Wend anymore.

 

 

With… End With

 

Applies to objects and  structures and apart from saving you typing, speeds up your code because the reference to the object or structure is only evaluated once.

 

For i = 1 to 100

    With StructureName(i)

        .StructureMember1 =

        .StructureMember2 =

        .StructureMember3 =

        .StructureMember4 =

    End With

Next I

 

 

Default Properties

 

If an object has properties then one of these is it’s default property. For example – the default property of a Text-box is it’s Text and that of a Check-box is it’s Value. Default properties are accessed faster than non-default properties; so

 

CheckBox = vbChecked

 

is faster than

 

CheckBox.Value = vbChecked

 
 

GoTo

 

Here the opposite to Option Explicit is true; do not use it,  do not use it, do not use it!

 

 

Error Handling

 

Sub Something

    On Error GoTo ErrorHandler

 

    [error tolerant code]

 

Exit Sub

 

ErrorHandler:

 

    MsgBox Err.Number & " " & Err.Description, vbCritical, "Error in Module Something"

 

End Sub

 

Now, what happens here when an error occurs? The customer sees a box telling him some error occurred; she doesn't know what the error is - some gibberish she can't make any sense of, so she clicks it away anyhow, it's bad news. And what's more - the application simply continues as if nothing had happened. But of course something has happened! A more or less large part of the error tolerant code in Sub Something was not executed because it went straight to the ErrorHandler , with unpredictable and probably fatal results if the application (and the customer using it) is allowed to continue. This has two or possibly three effects: It annoys the user, it embarrasses the author, and it could destroy valuable data. The fact that the author has produced a faulty product is embarrassing, that he is unable to tell the user in his own language what has happened is embarrassing and the fact that the application in it's unstable state is continuing and doing all kinds of unpredictable things may turn out to be very embarrassing.

 

Are we ourselves not annoyed by some obscure hex values in Micro$oft's Protection Error Message Box? What does it tell us? Is it our fault if this box pops up? No, of course not - it's all Mocr$oft's fault!! Same with your product... it's YOUR fault!

 

And if the code is correct, i.e. not producing any errors? Then the Error Handler is dead code! Didn't we all learn to avoid dead code? Do you not wish and hope that the Error Handler will never be executed? Then why is it there? If you do need a reaction to errors, then definitely not to your own ones. Your own errors should simply not be there. And if you tested your application thoroughly and properly then you can take the "risk" to go live without this kind of "Error Handler". No error handling will at least crash the application and prevent further damage - the application is faulty anyway and needs bug fixing.

 

Proper error handling should prevent the user from making mistakes and that is a quite more complicated task than writing three automated lines into each and every module, even into those where an error can definitely never occur.

 

If you have to use internal error handling for some reason (for example Micro$oft’s Common Dialog Box returns an Error when the user clicks Cancel ) then catch the  Error where it occurs:

 

On Error Resume Next ‘comment on the reason for entering an error tolerant code passage

 

    [some short(!) code passage where an Error is expected]

    e = Err ‘Err.Number is the default property of the Err Object

 

On Error GoTo 0 ‘turn error checking back on ! (This by the way is the only situation where  GoTo is unavoidable)

 

If e Then

    [do what has to be done in case the expected error has occurred]

  Else

    [do what is normally done without an error]

End If

 

 

If you have no plausible reason for an error tolerant code passage then there is nor reason to use it at all. And uncertainty about what could happen when your code is executed is definitely no plausible reason.

 
 

Encapsulation

 

The term encapsulation is often used interchangeably with the term information hiding - a software module hides information by encapsulating the information into a module or other construct (!)  which presents an interface. The most common use of information hiding is to hide the physical storage layout for data, but other types of encapsulation are equally important which have nothing to do with information hiding: I dare say that encapsulation in the narrower sense of the word goes down as far as code construct levels insofar as each construct level has one entry and one exit. A construct level is a stretch of code which is dependent on and encapsulated by code which surrounds it, for example a loop body which is surrounded by  Do - Loop or For – Next, or decision depend code surrounded by If – End If or Select – End Select. Or simply code surrounded by Sub – End Sub.

 

Imagine you have two nested loops, one within the other, each constituting a construct level,  and the outer loop should also terminate prematurely when the inner loop terminates prematurely for some reason – in other words you want to get out of both loops simultaneously. It has been suggested that  GoTo EndOfBothLoops  is the best solution or is at least a venial sin.  I personally do not agree – this is breaking the iron law of ‘one entry and one exit’ as the following example shows:

 

Private Sub BothLoops

 

  Dim i As Long, j As Long

 

    For i = 1 To m

        For j = 1 To n

            [loop body]

            If SomeCondition Then GoTo EndOfBothLoops

        Next j

    Next I

 

EndOfBothLoops:

 

    [more code]

 

End Sub

 

 

Now look at this:

 

Private Sub OuterLoop

 

  Dim i As Long

 

    For i = 1 To m

       InnerLoop

    Next i

 

EndOfBothLoops:

 

    [more code]

 

End Sub

 

 

Private Sub InnerLoop

   

  Dim j as long

   

    For j = 1 To n

        [loop body}

        If SomeCondition Then GoTo EndOfBothLoops   ‘well – you can’t !!!     and you shouldn’t either, even if you could

    Next j

 

End Sub

                               

 

Here both loops have a single entry and a single exit and are explicitly encapsulated -  therefore it is impossible to cross construct borders. In a good structured design you should not (willingly) cross construct borders. Think of something else instead.

 

 

Code Size

 

A module in excess of around 3,000 lines is indigestible, and even if I find something in one of the forums which I admire and would like to use in one of my projects, I personally would not use it because I cannot understand it and would not be able to fix it if it goes wrong.

 

And another thought is also important: Consider a code passage with only ten independent yes|no conditions. To test it thoroughly you will have to devise 1,024 test cases to test every possible path through that code. If however you split it into two parts, each having five such conditions then you only need 32 times 2 = 64 test cases. Quite a saving, isn’t it? I know, this is only a theoretical thought but it shows the tendency.

 

So keep your modules small and don’t try to create a Shmoo.

 

For Shmoon (plural of Shmoo) see: http://www.lil-abner.com/shmoo.html

 

 

The Stack

 

I promised earlier to say something about the stack. Now then, the stack is an area in memory set aside to save certain data in an unfair manner; not first come first serve but the opposite way: last come first serve. You can think of the stack as a pipe which is closed at one end and open at the other. You can get items from it only in reverse order of putting them in.

 

Have you ever wondered about the mechanism that VB uses to find back to the calling module after the called module has finished? Well – the call puts it’s return address on top the stack (called Push), and the return just picks up and discards the newest address from the stack (called Pop) and continues from there.

 

But that’s not all the stack is used for. When you call a module using parameters then these parameters are also put on the stack for the called module to find them. And that’s still not all: when a module declares local variables these are also arranged on the stack. You might say that each module gets a dynamically allocated stack frame upon (re-)activation and that stack frame stays active until the module returns; only then it is abandoned. By the way, the Error object uses these frames in reverse order  to find an active Error handler.

 

Now, what’s all this good for? Imagine you have a program running and at the same time you are typing a character on the keyboard. This will of course have to be handled by Windows immediately (before you type the next character) and the only way to do that is to interrupt your running program and handle the keystroke. Then after that your program will have to be resumed at exactly the spot where it was interrupted and with exactly the state it was in when it was interrupted. Ah… stack again – the address where it was interrupted is pushed on the stack and also the state your program was in at the time of interruption. But… if your program is just then monopolizing the CPU? It is interrupted anyway and the keystroke is processed, but the process the keyed character was meant for cannot be informed that is has a keystroke pending because you are just now monopolizing the CPU. So to be cooperative you have to give way (yield) every now and then and let others use the CPU for a while to handle their events. Does it sound familiar? Handle their events – do their events – DoEvents – and that applies to your own events as well! We’ll return to that topic when we talk about the Windows message system.

 

I just said that local variables (Dim) and call-parameters are also arranged on the stack. Now – this enables us to do something which seems strange at first but which is quite helpful in certain situations: a module can call itself. Why would we do that, you ask?

 

Imagine you have a savings account and you want to know what the final value of it is after say ten years. A first analysis shows that after 0 (zero) years the final value is the capital only without interest and in all other cases the final value is the previous year’s final value plus interest. Are you with me? If not read that last sentence again because it is important to understand the logic behind it.

 

Okay, then let’s try and design a function to do exactly that. It will of course need to know the capital, the interest rate and the number of years, and it returns a currency type final value:

 

    Public Function FinalValue (Capital As Currency, InterestRate As Single, Years As Long) As Currency

 

        If Years = 0 Then     ‘after zero years the final value is the capital only without interest

            FinalValue = Capital

          Else                ‘the final value is the previous years final value plus interest (we don’t know the previous year’s final value yet so we let this function work it out)

            FinalValue = FinalValue(Capital, InterestRate, Years – 1) * (1 + InterestRate / 100)

        End If

 

   End Function

       

Hu…??? Yes, that code represents exactly what we found in the analysis above. But does it work? Try it…

 

It’s called recursion and it can come in quite handy at times. There is only one thing to consider: there must be a condition when it stops calling itself and it must be working towards that condition.

 

 

Here’s another example for recursion:

 

The faculty of a number is the product (by multiplying) all natural numbers up to and including the number. It’s written with an exclamation mark: 4! is therefore 1 * 2 * 3 * 4 = 24. This is the rule but of course, as always, there is an exception to the rule: 0! = 1 (don’t ask me why, I’m far from being a mathematician).

Hah!  I found out why: the number of permutations of elements in a set equals  (NumberOfElements)! and an empty set with zero elements has one permutation.

 

The analysis therefore is as follows: If the number is 0 (zero) then it’s faculty is 1 and in all other cases the faculty is the result of multiplying the faculty of the preceding number by the current number;

for example    4! = 1 * 2 * 3 * 4

                          = (1 * 2 * 3) * 4

                          = 3! * 4

 

    Public Function Faculty (Number As Long) As Long

 

        If Number = 0 Then     ‘if the umber is 0 (zero) then it’s faculty is 1     

            Faculty = 1

          Else                 ‘the faculty is the result of multiplying the faculty of the preceding number by the current number

            Faculty = Faculty(Number – 1) * Number

        End If

 

   End Function

 

I know, these two examples could also be coded as a “normal” loop. But they make the principle of recursion clear and are definitely more elegant solutions. When the repetition does not occur at the end (as in these examples) but somewhere in the middle it is always worth considering recursion. Local variables will be preserved (they are on the stack) as this fragment shows:

 

 

    Private Function Recursion(…) as Type

 

      Dim i as Long

 

        If SomeCondition Then

            DoSomething 

           [No Recursion]

          Else

            For i = 1 To 10

                DoSomeOtherThing

 

                Recursion

 

                DoSomeMore

            Next i

        End If

 

    End Function

                                                                                                                                                

A new i will be used for each recursion level (remember the stack frame which is allocated upon module (re-)activation), and when the Recursion returns it will find the i of this recursion level as it was before. There is a practical limitation on the number of recursion levels by the stack size – when I remember correctly VB has a stack of 40 kBytes; when this is used up you’ll get a Stack Overflow  error.

 

Tables, Arrays

 

Sometimes you have more than one variable of the same kind; say you’re dealing with city names. All are strings and all those strings mean the same – the name of one particular city. Instead of allocating a string for each individual city you could pack all those strings into a table with so many items as there are cities.

 

You do that by defining the table first (when you use any kind of repetitive variables it is good practice to use the plural form of the variable name - not City but Cities; there’s more than one, so why not say so?)

 

    Private Cities(1 To 5000) As String

 

and you can then refer to each individual city by using an Index

 

    Cities(3768) 

 

refers to the 3768th city in that table.

 

The same is true when you write the following

 

    i = 3768

    Cities(i)

 

If you want to process all cities in that table you can do that in a loop where you vary the index from 1 to 5000. The  For..Next structure is most suitable for that:

 

    For i = 1 to 5000

        Do something with Cities(i)

    Next i

 

When you do not initially know how many items will be in a table you can also leave the number of items open and define that later while your code is executing:

 

    Private Cities() As String

 

    ReDim Cities(1 to 4978)

 

    ReDim Preserve Cities(1 To NumCities)

 

There is an  Array  type variable also which you could use instead of a table (in fact it’s a  Variant), and this even has a function to fill it all in one go.

 

  Dim Streets As Variant

 

    Streets = Array(“Baker Street”, “Oxford Street”, “Kings Road”, “Park Avenue”, “Rotten Row”)

 

and  Streets(2) refers to Kings Road because the index of an  Array is zero-based.

 

An  Array will accept all types of data, even other arrays, and on top of that data types can be mixed in one array:

 

    Misc = Array("London", "New York", 5, 72.34, Array(vbRed, vbGreen / 256 / 2, vbBlue / 256 / 256 / 3), Form1)

 

You can refer to  vbGreen for example by coding  Misc(4)(1) and code like the following is legal, although not very efficient:

 

    Misc(5).BackColor = RGB(Misc(4)(0), Misc(4)(1), Misc(4)(2))

    Misc(5).Print Tab(Misc(2)); Misc(0)

 

Regarding the efficiency of the previous code: At compile time VB does not know what kind of data types are stored in the array and in particular whether  Misc(5) can in fact have a  BackColor  at all or could be printed to. So these checks have to be delayed until the code is executed at run time. This mechanism is called Late Binding and it is quite inefficient. It’s okay if the code performed just once but it is definitely not recommended in a loop.

 

Strings in VB

 

There was a change from VB3 to VB4 and VB is now using what’s called a Basic String or BSTR for short. BSTRs are also used by COM, Automation, and Interop functions but unfortunately not by the API.

BSTRs have the following characteristics:

A BSTR is a composite data type that consists of:

·         Length Prefix

·         Data String

·         Terminator

Length Prefix:

·         A four-byte integer   

·         Occurs immediately before the first character of the data string

·         Contains the number of bytes in the following data string

·         Does not include the terminator

Data String:

·         Windows Platform: A string of unicode characters (wide characters, also known as  double byte characters).

·         Apple PowerMac: A single-byte string.

·         The string can contain multiple embedded null characters.

Terminator

·         Consists of two null characters (0x0000).

 

Most of this is transparent in VB, that is: you see just the data string. However in VB this concept has even been extended by a pointer; this pointer is a 32 bit value having the memory address where the actual string is stored:

 

Pointer

XXXXXX  ------------+      points to the first unicode  character

                    |

                                        |

                    |

                  length            V                termination

 BSTR       00000008   U l l i  0000

 

 

There are two undocumented functions in VB:  VarPtr and  StrPtr; VarPtr will return the memory address of the pointer and StrPtr will return the memory address of the string itself - that is the contents of VarPtr (in our example above the  xxxxxx).

 

For insiders: That’s the reason by the way why we have to pass strings by value when it comes to API calls; what we pass is in fact the value in the pointer – the address of the string. However, as we’re NOT passing the length, the called API function has to rely on the terminator only. If the string itself contains Chr$(0) then this will be mistaken for the terminator, so do not pass strings to an API function which containembedded Chr$(0)-characters.

 

A vbNullString is just a Pointer of zero and no BSTR, whereas “” is a Pointer to a BSTR-terminator with a preceding zero length prefix. They look almost the same in VB, the only way to differentiate is to call  StrPtr – if that returns zero then it’s a  vbNullString, otherwise it’s a “”. 

 

The  Len function returns the length prefix divided by two (remember – we are dealing with unicode characters (two bytes each, the length prefix however has the number of bytes).

 

 

  Dim a As String

 

    a = vbNullString

    Debug.Print “>”; a ; “<”, StrPtr(a)

 

    a = “”

    Debug.Print “>”; a ; “<”, StrPtr(a)

 

 

 

A canceled  InputBox  returns a  vbNullString – that’s is how you could differentiate between the different user responses; try this:

 

  Dim UserInput As String
 
    UserInput = InputBox("Please type in nothing or some text and click OK or Cancel", "Tri State")
 
    Select Case True
 
      Case StrPtr(UserInput) = 0
        MsgBox "You clicked Cancel"
 
      Case Len(UserInput)
        MsgBox "You typed """ & UserInput & """ and clicked OK"
 
      Case Else
        MsgBox "You typed nothing and clicked OK"
 
    End Select
 

 

To clarify things a little more try the following code:

 

Private Declare Sub PokeLong Lib "kernel32" Alias "RtlMoveMemory" _

                    (Destination As Any, _

                     Source As Any, _

                     Optional ByVal Length As Long = 4)

 

This declares an API prototype call which copies four bytes (a  Long,  32 bits)  from Source to Destination

 

 

  Dim a As String

  Dim b As String

  Dim lng As Long

 

    a = "Ulli"

    b = "Frederic Flintstone"

 

    Debug.Print Len(b); b

    PokeLong ByVal VarPtr(b), ByVal VarPtr(a) ‘copies the a-pointer to the b-pointer

    Debug.Print Len(b); b

 

    PokeLong lng, ByVal StrPtr(b) – 4 ‘that’s the length in bytes

    Debug.Print lng

   

    PokeLong lng, ByVal StrPtr(b) ‘and this is the first two Unicode chars

    Debug.Print Hex$(lng)

 

 

Why are we using  ByVal  sometimes and not always? This particular API call expects the addresses of source and destination, and  VarPtr  and  StrPtr  both return an address which we can use as it is – ByVal. On the other hand – the value of  lng  is undefined and we don’t want to use the value of  lng  but a reference to it, it’s address, to receive the moved bytes.

 

It seems as though we could duplicate strings just by copying the pointer; however when we do that (as in our example above) the BSTR “Frederic Flintstone” still exists but is orphaned now because it’s original pointer was overwritten and points to the BSTR “Ulli”. Therefore VB cannot find the Fred-BSTR anymore, probably with unpredictable consequences.

 

Collections

 

No, we are not talking about donating money for charity; a  Collection  in the context of this article is an object which is able to store an arbitrary number of items and also offers means to recover them.

 

The way the object does this is transparent to the user, but the object exposes a number of properties and methods which can be queried or used to achieve the above-mentioned goals.

 

An object must be instantiated, that is it must come to life so to say. In it’s dormant state it’s just an idea or a plan which defines what it looks like from outside and how things will be done internally, similar to the plan of a car; when you want to buy a  New car you would probably look at the pictures and specifications and then you order it and it is built for you. The emphasis here lies on the word  New.

 

Objects have a Type (which is called Class in this context),  similar to the type of variables, and this particular object is of class Collection, so obviously you will have to declare it as such:

 

  Dim Items As Collection (Items is called an object variable and refers to a particular object once that is instantiated. Important:  It is  NOT the object itself)

 

But defining it does not mean it is instantiated, until now it merely exists as an idea in it’s dormant state – or to stick to the previous example: the garage is there but there is no car in it yet . So we have to make it alive:

 

    Set Items = New Collection

 

The keyword  New  tells VB that we are not talking about an existing collection, but that we want a new one with no miles on the clock so to say. Items now has a reference to the instantiated object and you can obtain this reference by using an undocumented function ObjPtr. The reference points to the object’s virtual table or VTable; what that is will be discussed later.

 

So now the Collection object really exists and is ready to be used. How then can we use it? I said previously that it exposes a number of properties and methods which we can use to “make things  happen”. Here are the properties, a collection has two:

 

·         Count                                tells you how many items are stored in it.

 

·         Item(Index or Key)   lets you recover one item from the collection without removing it, the parameter tells the collection which item is meant.

 

Here are the methods:

 

  • Add OneItem [, Key, Before, After]    (the square brackets indicate that these parameters a optional)

 

  • Remove Index or Key

 

As there may be more than one collection in your program you will of course have to specify whose properties or methods are meant by putting the name and a dot in front of it, e.g.:

 

    Items.Add ..

 

 

Add     does what it’s name implies – it adds an item to the collection, the optional parameters further specify how and where the item should be added; without these parameters the  Item  is appended at the end and/or without a Key. You can add anything to a collection – data of any type and/or  objects of any class. The Key should be a String. The Index is automatically generated, starting with one (1).

 

 So this loop will address every item in a collection:

 

  Dim OneItem As [whatever data type is compatible with all items stored in the collerctio]

 

    For i = 1 To Items.Count 

        OneItem = Items.Item(i)   ‘copy the ith item from the collection into the local variable.

                                  ‘you can omit the Item property because that is the default property of a Collection    

        do something with OneItem

    Next i

 

Collections however incorporate a little counter inside which can be activated by the keyword  Each. This counter can also be used (instead of an index) to address each item in a collection and copy them to the variable  OneItem  one after the other:

 

  Dim OneItem As Variant ‘this must be a Variant type or an Object type

 

    For Each OneItem In Items

        do something with OneItem

    Next OneItem

 

 

Remove      removes one item from the collection and the parameter defines which item should be removed. If the removed item was somewhere in the middle the ensuing gap is closed.

 

Strangely Microsoft did not implement a .Clear method nor a .Replace method and no .Sorted property either. Also one cannot recover the  Key of an  Item. Clearing a collection of all items can simply be achieved by Setting  the collection to  New  again (if there’s only one reference to it, see Multiple References below). If there are multiple references to a collection you will have to clear the collection manually:

 

    With Items

        Do Until .Count = 0

            .Remove 1

        Loop

    End With

 

A little quiz for you: Why do we only  .Remove 1 (the 1st  Item) in this loop?

 

To replace an item you will have to  Remove and then  Add it:

 

  Dim OneItem As [whatever data type is compatible with all items stored in the collerctio]

 

    For i = 1 To Items.Count 

        OneItem = Items.Item(i) ‘copy the ith item from the collection into the local variable

                                ‘you can omit the Item property because that’s the default of a Collection: OneItem = Items(i)    

        do something with OneItem

        Items.Remove i

        Items.Add OneItem, Before:=i

    Next i

 

Using the  With .. End With  construct will greatly simplify your code and will also make it faster:

 

    With Items

        For i = 1 To .Count 

            OneItem = .Item(i)   ‘copy the ith item from the collection into the local variable

            do something with OneItem

            .Remove i

            .Add OneItem, Before:=i

        Next I

    End With

 

A little quiz for you: Why  Before? And what is this strange syntax with :=  (colon-equal)?

 

 

Now, what is the  Key good for if it cannot be retrieved? The  Key is an optional string value you can associate with an  Item  to identify it and you can use it just like you would use an index:

 

  Dim Salaries1 As Collection

 

    Set Salaries1 = New Collection

 

    Salaries1.Add 350000, Key:=“Ulli”

 

    Debug.Print Salaries1(“Ulli”) ‘Item property omitted this time and indexed by Key rather than index

 

 

You can also declare a Collection (or any other object ) As New

 

  Dim Employees As New Collection

 

But that does not (repeat NOT) mean the object is instantiated before you first use it – it’s instantiated on the fly so to say. The implication is clear – VB has to check on every access to the object what it is and whether or not it already exists – late binding sends greetings. So for the sake of efficiency – do not use this construct.

 

                                                                                                           

Multiple References

 

You can create another object variable which refers to the same object simply by

 

  Dim Salaries2 As Collection

 

      Set Salaries2 = Salaries1

 

This does not instantiate a new object, it merely creates another reference to the existing collection object and at the same time increases a counter inside the object by 1; this reference counter keeps track of the number of references the object has.

 

An object must be destroyed when it’s no longer needed. Now here’s the rule: An object is destroyed when the last object variable having a reference to the object is set to  Nothing  or goes out of scope. Normally you would destroy an object by

 

      Set Salaries1 = Nothing

 

This decreases the object’s reference counter. In the above example however you have two object variables referring to the same object and the object will still exist until you also

 

      Set Salaries2 = Nothing

 

and now the reference counter is zero which means that you have no means anymore to access the object and therefore it’s destroyed, i.e. removed from memory.

 

 

If the (only) object variable is local (that is: defined inside a module) that variable will be in the current stack frame (see The Stack above) and the object will be destroyed when the current stack frame including the object variable is abandoned on exit from the module.

 

 

A “little” problem exists:

 

Imagine you have declared two Collections in the declarations part

 

Private Companies As Collection

Private Employees As Collection

 

and later in code you would code this:

 

    Set Companies = New Collection

    Set Employees = New Collection

 

 

    For [each company]

 

        populating the Employees Collection

        .

        .

        Companies.Add Employees, Key:=CompanyName

 

        Set Employees = Nothing ‘hoping that would destroy the collection object and free the memory for the employees of the next company.

 

   Next [company]

 

But would that last statement really destroy the Employees collection object? The answer is no, because there’s another reference to the Employees collection in the Companies collection. So you would have to destroy the Companies collection first (or at least remove the Item from Companies which refers to Employees). That’s definitely what you don’t want after you’ve just added it.

 

Even worse: when you have self-, crossing- or circular references you cannot destroy any of the objects which participate in the reference chain unless there is a possibility to break that chain somewhere and tidy up in reverse from there.

 

One last mystery for you to solve:

 

    Private Sub Mystery

 

      Dim Computers As Collection

 

        Set Computers = New Collection

 

        Computers.Add Computers

 

    End Sub

A little excursion to

 

Data Structures

 

Tables or Arrays

 

The term table is used here to differentiate between tables and an intrinsic VB-function called Array which generates a table.

 

Tables are typically unordered sets of similar items arranged in one or more dimensions. Each item can be uniquely addressed by it's index(es), also called subscripts in some programming languages. VB allows a de facto unlimited number of dimensions (80 I believe but I’m not sure) and also permits re-dimensioning tables with or without keeping the current contents. VB-tables are realized by a mechanism called Safe Arrays; safety here means that a table is safeguarded against illegal indexes which would otherwise address non-existent elements beyond the bounds of the table. You can also bias the indexes – meaning that indexes do not have to start at 1.

 

Example:

 

    Public ChessSquares (1 To 8, 1 To 8)

 

It is good practice to use the plural form of the items in the table to name the table and indicate that there are more than one item (hoping you’ll never have to declare a table of Sheep #grin# )

 

    ChessSquares(1, 5) therefore might be the square for the white king.

 

 

Lists

 

are sets of items arranged in a one-dimensional order of predecessor-successor. Each item except the first has exactly one predecessor and each item except the last has one successor. The order is by position.

 

 

Linked Lists

 

same as lists except that each item carries one (or two) pointers which point to it's successor and / or predecessor; the order needs therefore not be positional but  is established by the pointers. A header is necessary which points to the first / last (or both) items. A VB – Collection Object is arranged this way with non-circular forward and reverse linking of items. Linked lists can be arranged in a circular fashion - the last item in a chain then points back to the first instead of indicating end-of-chain with a pointer of zero. If the list is double-linked then the fist item also points to the last. A circular linked list therefore has no real beginning.

 

My ‘Sort Without Sorting’ ( http://www.planet-source-code.com/vb/scripts/ShowCode.asp?txtCodeId=55546&lngWId=1  ) for example uses multiple forward or reverse linked lists (called Chains there) and multiple headers (called Anchors).

 

 

Queues

 

are lists where items are added at the back and removed from the front, they are also called FIFO (first in, first out) structures. How the FIFO-mechanism is arranged is left to the programmer's imagination, but remember that items can arrive faster than they can be handled, so a queue may be growing. If the opposite is true then a queue can shrink to zero items. I remember when I made a semi-automatic taxi dispatcher software that empty queues caused some problems.  A forward circular linked list of sufficient size is one choice to realize a queue. You would have an input pointer which points to the next free slot (item) and an output pointer which points to the next ‘un-free’ slot. But keep in mind that the output pointer must not ‘overtake’ the input pointer (and vice versa) This condition should in fact cause the dispatcher to wait – i.e. if there is no ‘un-free’ slot (no customer waiting).

 

 

Stacks

 

Similar to queues with one exception: items are added at and removed from the same end, so they are also called LIFO (last in, first out) structures. The 386-hardware (your CPU) has a number of registers and instructions to support such a stack. Putting an item on top of the stack is called Pushing and taking the top item off the stack is called Poping.

 

 

Trees

 

...they are the only trees which have the root at the top.

 

 In this particular arrangement items are called (or rather reside in) nodes and the links are called edges (I think). Trees are similar to (normally non-circular) linked lists except a node can have more than one successor. The first node (the one with no predecessor) is called the root and nodes with no successor are called leaves. Nodes with successors are also called parent-nodes and their successors child-nodes. If the links to a parent's children are ordered then it's called an ordered tree and if each parent has at most two children then it's called a binary tree. Children can link back to their parent; this enables us to traverse a tree in both directions – from the root to a leaf and from a leaf to the root. Some compression algorithms (for example Huffman) use binary trees.

 

Trees have a recursive structure and are therefore self-similar: each node can also be looked at as the root of a sub-tree.

 

I have a suspicion that this terminology was invented by some important professor who thought that a new terminology would make things look a lot more difficult than they are; I further suspect that this particular professor was abstaining from alcohol because otherwise he might have called them grapes and stalks, and the leaves berries; at least this would not make them appear upside down.

 

 

The API (Application Program Interface)

 

describes a standard set of rules for accessing a massive amount of subroutines and functions (collectively called Methods) built into the Operating System. We will constrain ourselves to Windows here, but other OS’s  are similar.

 

Each Method has a Name, resides in a File, expects from zero to n  Parameters, will execute a specific Task and most of them return a Value indicating success or failure, although some of them return a Result which is both – a legal result of the task or a failure indicator.

 

In VB (which has no header files like for example C#) you must declare a prototype of the function call so that the compiler can check that you used it correctly. This prototype has an identical meaning for VB as a  Sub / Function  you have written yourself – Scope, Name, Parameters and Return-Value – with one exception: since it is  External you must also state the file where it can be found. Now let’s declare a prototype:

 

Public Declare Function SendMessage Lib "user32" Alias "SendMessageA" (ByVal hWnd As Long, ByVal wMsg As Long, ByVal wParam As Long, lParam As Any) As Long

 

and see what we got here. Obviously the scope is  Public, meaning that it can be used by all members of your program. Next – it’s a  Function  and we expect it to return a value. The Name we are going to use is  SendMessage  although the external name  (Alias) is  SendMessageA. Aliases are necessary when the external name either does not conform to the naming rules in VB or has been changed by M$ but we still want to use the old name. This particular function expects four parameters (all of which are of type Long - important) and their names (not important) are placeholders for the names we’re going to use when we actually make the call (they have the role of reminders while typing the call because VB will display them while you type) and finally – the type of the function is  Long – indicating that we expect a  Long as the returned value.

 

And now the specifiers  ByVal, ByRef (which is the default specifier) and  As Any:

 

As I explained earlier: parameters are pushed on the stack and there are two choices what to push – either the value of the parameter itself or a reference to the parameter, i.e. the memory address where the called method can locate it. ByVal indicates that the called method expects the parameter’s value on the stack and ByRef indicates that the called method expects the  memory address of the parameter on the stack. The former of course is a little faster because the called method saves the step of fetching the parameter from the indicated memory address.

 

And finally  As Any – this does not indicate the called method accepts anything for that parameter; it merely indicates that you do not want to specify yet, how you will arrange for that parameter to have a legal value; SendMessage expects a  Long  as the forth parameter but that can be anything: a ByVal-value, the address of a string or data structure or even an object pointer or a window handle, and you must specify what it is when coding the call itself, as long as it is a  Long with the expected meaning it’s okay.

 

For Insiders

Parameters are pushed onto the 386-stack in reverse order – rightmost parameter first - and the stack “grows” towards lower memory-addresses, so the first or leftmost parameter is at the lowest stack address, just one dword  (four bytes) beyond the return address, when the called method is given control. When you write a method in Assembler the usual prolog is as follows:

 

Push EBP

Mov  EBP,ESP     ;save stack pointer in EBP after pushing EBP

 

Push 0           ;once for per 4 bytes of local zero-initialized store

(alternative: Sub ESP,[number of bytes for local un-initialized store])

 

Push [other registers which you are going to use and which must be preserved]

 

The stack now looks like this:

 

  • other registers
  • local variable3
  • local variable2
  • local variable1

·         EBP

  • return address
  • param1
  • param2
  • param3 (remember – this was pushed first)
  • etc

 

the parameters therefore are at  [EBP] + 8..12..16.. etc  now, and the local variables are at [EBP] – 4..8..12…etc

 

[your code]

 

and this is the usual epilog:

 

Pop  [preserved registers in reverse order of pushing]

Mov  ESP,EBP     ;abandon local variables

Pop  EBP

Mov  EAX,Result  ;return result in EAX

Ret  [number of additional stack bytes (the parameters) to abandon]

 

 

By the way – there is one misleading quirk in VB – strings are usually passed  ByVal  although the called method expects the address of the string (a BStr to be specific, see Strings in VB) on the stack. What is passed (by value) is VB’s pointer the string and that is usually exactly what the called method expects – the memory address of the string.

 

Why is the string (the characters which constitute the string) not pushed directly onto the stack?

 

1.       The stack is 32 bit-oriented with fixed length entries whereas a string is of variable length

2.       The string is already there and we need not copy it for the called method to find it, we just tell it where the string is

 

Some API methods (for example those returning values from the registry) return a string and you have to provide the space for it,  and pass that to the called method. Since you usually don’t know beforehand how long the string is going to be some methods are polite and let you ask them first how much space they need; others just have a parameter telling them how much space you have provided and will truncate the string if that’s not enough; for those it’s a good policy to provide a loooong string of say 1024 chars. The returned value then tells you how long the string actually is upon return from the method – but keep in mind that theyusually do not return BSTRs but the C# flavor of a string where the trailing zero is included in the length count.

 

Here is an example how to deal with those:

 

    TmpString1 = String$(1024, 0)  ‘make room to receive a string       

    k = Len(TmpString1)            ‘get maximum length of string to pass to API method     

   

    If RegQueryValueEx(i, Fontface, REG_OPTION_RESERVED, j, ByVal TmpString1, k) = ERROR_NONE Then

 

         ‘this has put the requested string into the space provided by us and has also put the string’s length into k

         TmpString1 = Left$(TmpString1, k + (Asc(Mid$(TmpString1, k, 1)) = 0))               

 

      Else 'NOT REGQUERYVALUEEX(I,...

 

         ‘something went wrong with the call so we use the default

         TmpString1 = DefaultFontName               

 

    End If

 

What we do here if there was no error is this: we look at the end of the returned string ( using the returned length  k ) and truncate one additional char if that’s the trailing zero.

 

Chaining API Calls

 

Some method are written to work together – that is - one expects the return value of the other, usually a color or a  handle to some stock object like cursors or  icons and such. You can chain those calls by passing the return value of the first immediately to the second. Here’s an example:

 

    hBrush = CreateSolidBrush(GetSystemColor(vbButtonFace))

 

This will first call  GetSystemColor which returns (in our example) the color of the button face and then pass that directly to  CreateSolidBrush  which in turn creates a brush of that color and returns the handle of the created brush.

 

One other thing is worth mentioning: When you create objects within the OS (like painting brushes, fonts, regions, or contexts etc) these are of a permanent nature and you must delete them when you don’t need them anymore. They take up memory and Windows will not delete them automatically when your program finishes but will leave them as they are – residing in memory for no one to use. This behavior is called memory leak and is quite common among unexperienced programmers; you’ll end up with your memory full of un-used Windows objects and finally run out of memory.

 

Therefore:

 

       DeleteObject hBrush

 

 

And one final thought: When a called method generates an error deep inside Windows’ intestines it will usually crash the application or Windows or both, or at least leave memory leaks. It will definitely not be passed to VB’s error handling mechanism, so writing error tolerant code around API calls (see Error Handling) hoping that VB will catch an error generated within the OS is futile wishful-thinking. Just let it crash – there’s nothing sensible you could do now anyway but restart Windows and fix your code.
